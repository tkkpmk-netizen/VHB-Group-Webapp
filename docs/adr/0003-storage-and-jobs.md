# ADR 0003: S3 Object Storage and PostgreSQL Durable Jobs

Status: Accepted  
Date: 2026-07-02

## Context

Documents, imports, designs, thumbnails, builds, exports, and deployment
artifacts require binary storage and asynchronous execution. Storing bytes in
PostgreSQL or running long work in FastAPI request handlers would make both
database operations and API latency unpredictable.

## Decision

### Object storage

- Store asset metadata in PostgreSQL and bytes through an `ObjectStorage`
  interface.
- Use S3-compatible APIs; MinIO is the local implementation.
- Browsers upload and download through short-lived presigned URLs.
- Object keys are workspace-prefixed and generated by the server.
- An asset is usable only after object existence and size verification.

### Durable jobs

- Use a PostgreSQL queue while the platform remains a modular monolith.
- Workers claim work with `FOR UPDATE SKIP LOCKED`.
- Jobs have explicit state, attempts, retry backoff, a worker lease, and optional
  workspace-scoped idempotency keys.
- Request handlers enqueue; the standalone worker executes registered handlers.
- Payloads and results are JSONB, but job types and handlers remain explicit.

## Consequences

- Local development needs PostgreSQL, MinIO, one API process, and optionally one
  or more worker processes.
- Worker scale is horizontal without adding Redis/RabbitMQ at this stage.
- PostgreSQL is not intended to remain the queue at very high throughput. Revisit
  a dedicated broker when queue contention or scheduling volume justifies it.
- Presigned URLs keep large byte streams out of FastAPI.

## Operations

```bash
cd docker && docker compose up -d db minio minio-init
cd backend && uv run python -m alembic upgrade head
cd backend && uv run python -m app.worker
```
