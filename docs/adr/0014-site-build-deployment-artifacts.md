# ADR 0014: Site Builds Produce Object-Storage Deployment Artifacts

Status: Accepted  
Date: 2026-07-07

## Context

DP1-DP4 created site resources, public data bindings, a Web Designer, and design
import source. DP5 needs a production boundary that publishes build artifacts
without treating editor source JSON as the visitor runtime.

## Decision

Add `SiteDeployment` as the build/deployment record and use durable jobs for
site builds.

- Admins call `POST /sites/{site_id}/deployments`.
- The API creates a `SiteDeployment` with a monotonically increasing version and
  queues a `site.build` job.
- The worker builds published `SitePage` sources into one HTML artifact.
- The artifact is stored in object storage through the existing `Asset` model.
- Public render endpoints serve only the latest ready deployment for a
  published site.
- `data-vhb-binding` markers are hydrated by a small runtime script that calls
  the restricted public binding API from DP2.

## Rationale

This keeps editor source, build status, and visitor artifacts separate:

- `SitePage.content` remains editable source.
- `SiteDeployment` records build history and status.
- `Asset` owns generated bytes in object storage.
- The public runtime never calls admin APIs.

The MVP build artifact is intentionally simple HTML/CSS/JS. It is enough to
prove publishing and database-connected pages without introducing a heavy
server-side browser renderer.

## Consequences

- DP6 can add domains, environments, rollback, and cache invalidation using
  `SiteDeployment.version` and artifact asset ids.
- Build failures are retryable through the existing job queue.
- More advanced component interactivity should be compiled by future build
  steps, not embedded in `SitePage.content`.
- Generated HTML is cacheable for short periods, while dynamic data remains
  fetched through public binding APIs.
